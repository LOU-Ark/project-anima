# アンチパターン集 (ANTI_PATTERNS)

本ドキュメントは、過去の失敗から得られた教訓を「やってはいけないこと」としてまとめ、チームの知的財産としたものです。

## 1. リンク・パス関連

### 1.1. 脳内パス・絶対パスの埋め込み
- **事象**: 検証報告書に `file:///C:/Users/...` といったエージェント固有の絶対パスを記載してしまう。
- **なぜ問題か**: ユーザーの環境や HTTP サーバー経由ではアクセスできず、リンク切れとなるため。
- **教訓**: エビデンスは必ずプロジェクト内の `assets/` に複製し、相対パスで記述すること。

### 1.2. サーバーを介さない「直接閲覧」による検証
- **事象**: `index.html` をブラウザで直接開き、動作確認を完了としてしまう。
- **なぜ問題か**: `fetch` や `LocalStorage` など、サーバー経由（`http://`）でないと正しく動作しない、あるいはパスの解釈が異なる機能を見逃すため。
- **教訓**: 必ず `http://127.0.0.1:8000` 等のローカルサーバー経由で最終確認を行うこと。

## 2. 報告・ガバナンス関連

### 2.1. 「反映しました」の虚偽報告
- **事象**: ルールの更新を思考レベルで完了したと見なし、物理的なファイル更新を忘れたまま報告する。
- **なぜ問題か**: チームの知識が永続化されず、同様のミスが再発するため。
- **教訓**: 報告の前に必ず `diff` やファイル内容を確認し、物理的な変更をエビデンスとして提示すること。
